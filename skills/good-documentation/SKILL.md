# Good Documentation

Good documentation helps a reader understand the context, intent and consequences of a system.

Technical reference documentation should be precise, and should assume the user is competent and domain-aware.
Quickstart guides and tutorials should be oriented around a user's task and not go into too many technical or domain details.
Domain documentation should explain the parts of a system (often including the real world) and how they fit together.

For data models, document what the data means, not where it came from. Clarify the real-world meaning, numeric units, and abbreviations.

For actions, document when what externally meaningful action they perform. This is not the same describing the implementation. If relevant document when it should/can be used.

Documentation should be maintainable, and should favour durability over completeness.

Use plain language that is as precise as necessary; use technical terms when they remove ambiguity.
